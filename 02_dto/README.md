# Module 2 — DTOs and data layers

## 📌 What you'll learn in this module

- Why you should split data into layers (Transport / Domain / Persistence)
- How to do mapping between layers correctly
- Partial Update — updating only the fields that were provided
- Validation at the DTO level vs the Domain level
- Why you can't "share" one struct everywhere
- Senior developer best practices

---

## 🧒 DTO explained like a school form

Imagine your school has a form for a new student.

The student writes only:

```text
name: Ali
age: 15
class: 9A
```

But the school system also stores:

```text
student_id: generated by school
status: active
created_at: current time
internal_note: visible only to school workers
database_version: used only by the database
```

Should the student fill all of that? No.

The student should not choose their own `student_id`. They should not set their own `status`. They should not see `internal_note` or `database_version`.

That is the whole idea of DTOs:

> A DTO is a small struct made for one boundary: input from user, output to user, or communication with another service.

## 🪜 Step-by-step: why one struct becomes a problem

### Step 0: one struct for everything

At first, this looks simple:

```go
type Student struct {
    ID              string `json:"id" db:"id"`
    Name            string `json:"name" db:"name"`
    Age             int    `json:"age" db:"age"`
    Class           string `json:"class" db:"class"`
    Status          string `json:"status" db:"status"`
    InternalNote    string `json:"internal_note" db:"internal_note"`
    DatabaseVersion int    `json:"database_version" db:"database_version"`
}
```

But this struct is dangerous if you use it everywhere.

If it is used as an API request, the user can send:

```json
{
  "id": "student-admin",
  "name": "Ali",
  "age": 15,
  "class": "9A",
  "status": "graduated",
  "internal_note": "change my grades",
  "database_version": 999
}
```

That is not okay. The user is controlling fields that only the server or database should control.

### Step 1: create a request DTO

For input, keep only what the user is allowed to send:

```go
type CreateStudentRequest struct {
    Name  string `json:"name"`
    Age   int    `json:"age"`
    Class string `json:"class"`
}
```

Now the user cannot send `ID`, `Status`, `InternalNote`, or `DatabaseVersion`, because those fields do not exist in the request DTO.

### Step 2: create a domain model

Inside the application, use a clean business object:

```go
type StudentDomain struct {
    ID        string
    Name      string
    Age       int
    Class     string
    Status    string
    CreatedAt time.Time
}
```

This is the object your service works with. It has business fields, but no `json` tags and no `db` tags.

### Step 3: create a response DTO

For output, return only what the user should see:

```go
type StudentResponse struct {
    ID     string `json:"id"`
    Name   string `json:"name"`
    Age    int    `json:"age"`
    Class  string `json:"class"`
    Status string `json:"status"`
}
```

Notice: `InternalNote` and `DatabaseVersion` are still hidden.

### Step 4: create a database model

For the database, use a struct that matches the table:

```go
type StudentRow struct {
    ID              string    `db:"id"`
    Name            string    `db:"name"`
    Age             int       `db:"age"`
    Class           string    `db:"class"`
    Status          string    `db:"status"`
    CreatedAt       time.Time `db:"created_at"`
    InternalNote    string    `db:"internal_note"`
    DatabaseVersion int       `db:"database_version"`
}
```

This struct is only for the repository/database layer.

### Step 5: map between them

Mapping means copying safe fields from one shape to another:

Input DTO -> Domain:

```go
func StudentRequestToDomain(req CreateStudentRequest) StudentDomain {
    return StudentDomain{
        Name:      req.Name,
        Age:       req.Age,
        Class:     req.Class,
        Status:    "active",
        CreatedAt: time.Now(),
    }
}
```

The user gives `Name`, `Age`, and `Class`.

The server decides `Status` and `CreatedAt`.

Domain -> Response DTO:

```go
func StudentDomainToResponse(student StudentDomain) StudentResponse {
    return StudentResponse{
        ID:     student.ID,
        Name:   student.Name,
        Age:    student.Age,
        Class:  student.Class,
        Status: student.Status,
    }
}
```

The response returns useful public fields, but still hides `InternalNote` and `DatabaseVersion`.

Domain -> DB row:

```go
func StudentDomainToRow(student StudentDomain) StudentRow {
    return StudentRow{
        ID:              student.ID,
        Name:            student.Name,
        Age:             student.Age,
        Class:           student.Class,
        Status:          student.Status,
        CreatedAt:       student.CreatedAt,
        DatabaseVersion: 1,
    }
}
```

The DB row gets database-specific fields. The user never sees this shape.

That is why mapping is useful: it is the place where you control what is accepted, generated, hidden, and returned.

### Step 6: the full simple flow

```text
1. User sends JSON
       |
       v
2. CreateStudentRequest
       |
       v
3. StudentDomain
       |
       +--> StudentRow      -> save to database
       |
       +--> StudentResponse -> send back to user
```

In code, a service flow can look like this:

```go
func CreateStudent(req CreateStudentRequest) StudentResponse {
    student := StudentRequestToDomain(req)

    student.ID = "generated-by-server"

    row := StudentDomainToRow(student)
    _ = row // repository would save this

    return StudentDomainToResponse(student)
}
```

This is the important part: the request, response, domain model, and database row are connected, but they are not the same struct.

### Tiny rule

Use different structs for different jobs:

```text
CreateStudentRequest  -> input from user
StudentDomain         -> business logic
StudentResponse       -> output to user
StudentRow            -> database storage
```

Same real-world student. Different shapes for different jobs.

## ❓ Why split data into layers?

Short answer: because the **client**, the **business logic**, and the **database** do not need the same shape of data.

Think about a simple order API.

When a client creates an order, they should send only this:

```json
{
  "customer_id": "cust-123",
  "amount": 100
}
```

They should **not** send this:

```json
{
  "id": "ord-999",
  "customer_id": "cust-123",
  "amount": 100,
  "status": "confirmed",
  "created_at": "2026-05-19T10:00:00Z",
  "deleted_at": null,
  "version": 7
}
```

Why not?

- `id` must be generated by the server.
- `status` must be controlled by business rules.
- `created_at` must come from the server clock.
- `deleted_at` is an internal database field.
- `version` is a database locking detail.

That is why we create a small input DTO:

```go
type CreateOrderRequest struct {
    CustomerID string  `json:"customer_id"`
    Amount     float64 `json:"amount"`
}
```

And a separate response DTO:

```go
type OrderResponse struct {
    ID         string  `json:"id"`
    CustomerID string  `json:"customer_id"`
    Amount     float64 `json:"amount"`
    Status     string  `json:"status"`
    CreatedAt  string  `json:"created_at"`
}
```

The response includes fields the client may read, but still hides database internals like `deleted_at` and `version`.

## 🧠 The simplest mental model

DTO means **Data Transfer Object**.

It is not your business object. It is not your database row. It is just the shape of data crossing a boundary.

```
Outside world          Your service              Database
-------------          ------------              --------
JSON request   ─────▶  Domain model   ─────▶     DB row
DTO                    business rules            storage shape

JSON response  ◀─────  Domain model   ◀─────     DB row
DTO                    business rules            storage shape
```

Use a DTO every time data enters or leaves your service.

## 🍱 Simple example: one order, three shapes

### 1. Request DTO: what the client is allowed to send

```go
type CreateOrderRequest struct {
    CustomerID string  `json:"customer_id"`
    Amount     float64 `json:"amount"`
}
```

This is intentionally small. The client says: "I want to create an order for this customer with this amount."

### 2. Domain model: what the business code works with

```go
type OrderDomain struct {
    ID         string
    CustomerID string
    Amount     float64
    Status     OrderStatus
    CreatedAt  time.Time
    UpdatedAt  time.Time
}
```

The domain model contains the real business state. It has no `json` tags and no `db` tags. It belongs to your service logic.

### 3. DB model: what the database stores

```go
type OrderRow struct {
    ID         string     `db:"id"`
    CustomerID string     `db:"customer_id"`
    Amount     float64    `db:"amount"`
    Status     string     `db:"status"`
    CreatedAt  time.Time  `db:"created_at"`
    UpdatedAt  time.Time  `db:"updated_at"`
    DeletedAt  *time.Time `db:"deleted_at"`
    Version    int        `db:"version"`
}
```

The database row can contain technical fields. Those fields should not leak into the API.

## 🚨 What goes wrong without DTOs?

### Problem 1: the client can set fields it should not control

```go
type Order struct {
    ID     string `json:"id"`
    Status string `json:"status"`
}
```

If this struct is used for input, a client can send:

```json
{
  "id": "fake-id",
  "status": "confirmed"
}
```

Now the client is trying to do the server's job. A create DTO prevents this because it simply has no `ID` or `Status` field.

### Problem 2: internal fields leak to users

```go
type Order struct {
    CustomerID string  `json:"customer_id"`
    CostPrice  float64 `json:"cost_price"`
}
```

If this struct is returned directly from an API, users may see your internal cost price. A response DTO prevents this by exposing only safe fields.

### Problem 3: API changes accidentally break the database

If one struct has both tags:

```go
type Order struct {
    CustomerID string `json:"customer_id" db:"customer_id"`
}
```

Renaming the API field can tempt you to touch the same struct used by the repository. Separate DTO and DB structs keep these changes independent.

## 🔄 Mapping: the small price you pay

DTOs need mapping. That is good. Mapping is the moment where you clearly decide what crosses each boundary.

```go
func RequestToDomain(req CreateOrderRequest) OrderDomain {
    return OrderDomain{
        CustomerID: req.CustomerID,
        Amount:     req.Amount,
        Status:     OrderStatusPending,
        CreatedAt:  time.Now(),
        UpdatedAt:  time.Now(),
    }
}
```

Notice what happens here:

- The client provides `CustomerID` and `Amount`.
- The server chooses the initial `Status`.
- The server sets timestamps.
- Internal DB fields are not involved.

That is the main value of DTOs: each layer gets only what it should know.

---

In a real application the same data looks different at different levels:

```
HTTP request         →  Business logic     →  Database
CreateOrderRequest      Order (domain)        OrderRow
{                       {                     {
  customer_id: "x"        ID: "ord-123"         id: "ord-123"
  amount: 100             CustomerID: "x"       customer_id: "x"
}                         Amount: 100           amount: 100
                          Status: pending       status: "pending"
                          CreatedAt: ...        created_at: ...
                          UpdatedAt: ...        updated_at: ...
                        }                       deleted_at: NULL
                                                version: 1
                                             }
```

### If you use one struct for everything — problems:

```go
// ❌ One struct for everything — BAD:
type Order struct {
    ID         string     `json:"id" db:"id"`
    CustomerID string     `json:"customer_id" db:"customer_id"`
    Amount     float64    `json:"amount" db:"amount"`
    Status     string     `json:"status" db:"status"`
    DeletedAt  *time.Time `json:"deleted_at" db:"deleted_at"` // internal field!
    CostPrice  float64    `json:"cost_price" db:"cost_price"` // secret field!
    Version    int        `json:"version" db:"version"`       // a DB technical artifact
}
```

**Problems:**
1. `deleted_at` and `cost_price` leak into the API response — **internal data leak!**
2. You changed the API (renamed a field) → you broke the DB schema
3. The client can send `version` → **vulnerability!**
4. Domain logic is mixed with HTTP logic and DB logic

## 🏗️ Three data layers

```
┌─────────────────────────────────────────────────────┐
│                  HTTP / gRPC                        │
│  CreateOrderRequest  ←→  OrderResponse              │
│        (json tags)                                  │
└─────────────────────┬───────────────────────────────┘
                      │  mapper.RequestToDomain()
                      ▼
┌─────────────────────────────────────────────────────┐
│                  DOMAIN                             │
│               OrderDomain                           │
│           (no tags, business only)                  │
└─────────────────────┬───────────────────────────────┘
                      │  mapper.DomainToDB()
                      ▼
┌─────────────────────────────────────────────────────┐
│                  DATABASE                           │
│                  OrderRow                           │
│             (db tags, soft delete, version)         │
└─────────────────────────────────────────────────────┘
```

| Layer           | Struct                                  | Where it lives | Tags | Who sees it               |
|-----------------|-----------------------------------------|----------------|------|---------------------------|
| **Transport**   | `CreateOrderRequest`, `OrderResponse`   | HTTP handler | `json:` | API client                |
| **Domain**      | `OrderDomain`                           | Service | no tags | Only service code         |
| **Persistence** | `OrderRow`                              | Repository | `db:` | Only the repository       |

                      ▼
                 OrderDomain  ←─────────────────────── DB SELECT

## ✏️ Partial Update Pattern

When the client wants to update only some of the fields — use a pointer:

```go
type UpdateOrderRequest struct {
    Amount *float64 `json:"amount,omitempty"`  // pointer — field is optional
    // If nil — the client did not pass the field → don't update
    // If not nil — the client sent a value → update
}

func ApplyUpdateRequest(order *OrderDomain, req UpdateOrderRequest) {
    if req.Amount != nil {
        order.Amount = *req.Amount  // dereference the pointer
    }
    order.UpdatedAt = time.Now()  // always refresh the timestamp
}
```

Why is this better than just `Amount float64`?

```go
// The problem with float64 (no pointer):
req := UpdateOrderRequest{Amount: 0}
// Amount == 0 — did the client want to zero it? Or just didn't send the field?
// Unknown!

// With a pointer it's unambiguous:
req1 := UpdateOrderRequest{Amount: nil}    // field not passed
req2 := UpdateOrderRequest{Amount: &zero}  // explicitly zeroed
```

## ✅ Validation: DTO vs Domain

There are two validation levels and both are needed:

```go
// Level 1: DTO validation (syntax)
// Check that data arrived in the correct format
func (req CreateOrderRequest) Validate() error {
    if req.CustomerID == "" {
        return errors.New("customer_id is required")
    }
    if req.Amount <= 0 {
        return fmt.Errorf("amount must be positive, got %.2f", req.Amount)
    }
    return nil
}

// Level 2: Domain validation (business rules)
// Check business invariants
func (o *OrderDomain) Validate() error {
    if o.Amount > 1_000_000 {
        return errors.New("order amount exceeds maximum limit")
    }
    if len(o.CustomerID) < 3 {
        return errors.New("invalid customer ID format")
    }
    return nil
}
```

## 💡 Why this is critical in microservices

In microservices, services communicate via gRPC or REST.
Each service has its own "truth" about the same entity:

```
OrderService                    InventoryService
{                               {
  order_id: "123"                 order_id: "123"
  customer_id: "456"              items: [...]
  total_amount: 100               warehouse_id: "wh-1"
  status: "confirmed"           }
  payment_method: "card"
}
```

A DTO is a **public contract between services**. It must be:
- **Versioned** (`/api/v1/orders`, `/api/v2/orders`)
- **Backward compatible** (you can't delete fields)
- **Minimal** (only what the consumer needs)

## 🏆 Senior developer's top rules

> **1. Never return a domain model directly in an HTTP response.**
> Always map through a DTO. Otherwise you risk leaking internal fields.

> **2. Never accept ID, CreatedAt, Status from the client on create.**
> These fields are generated by the server. The client must not set them.

> **3. Use pointer fields for partial updates.**
> `*float64` ≠ `float64` — it's a semantic "passed/not passed" distinction.

> **4. Validate DTOs on input, business rules in the domain.**
> Two layers of defense are better than one.

> **5. Format time as a string in DTOs, use time.Time in the domain.**
> Clients prefer strings; inside your code — time.Time.
